# Clavus Phase 2: Watch + Render + Ready

**Goal:** Make clavus immediately useful — auto-snapshot on save, and export stems for review.

## Current State

Phase 1 complete and pushed:
- `.als` parser (Ableton 10+/Live 9) — tracks, devices, markers, BPM, color, freeze/mute/solo
- Git-like content-addressed storage (SHA256, objects/refs/index)
- Snapshot/diff engine with formatted output
- CLI: `init`, `snapshot`, `log`, `diff`, `status`
- Cue system (CRUD, timeline-anchored, render-to-markers)
- Test suite covering parser, store, CLI, cues

## What to Build (in priority order)

### 1. `clavus watch` — File watcher daemon

Auto-snapshot the project every time the `.als` file is modified on disk. This is the killer feature — every time you hit Cmd+S in Ableton, clavus records the state.

- Uses `file watchdog` (or `polling` since .als files are gzip-compressed XML, file modification time is sufficient)
- Detects change via mtime change on the `.als` file
- Takes a snapshot with auto-generated message like "Auto-snapshot @ 2026-05-01 10:17"
- Daemonizes or runs as a foreground process (user choice)
- Debounce writes (Ableton may write multiple times on save) — 30s cooldown
- Skipped if no changes detected since last snapshot

**Files:** `clavus/watch.py` (new), `clavus/cli.py` (add `watch` command)

### 2. `clavus render` — Stem export

Render individual tracks from an Ableton project as audio files. This requires interfacing with Ableton's Live Object Model (LOM) via OSC/MIDI or the Live Control Surface API.

**Approach A (Recommended): PyLive + OSC bridge**
- Use `pylive` or the `python-ableton-osc` bridge to connect to Live remotely
- Live must have the control surface script/device running
- Commands: arm track → set render locators → export → collect

**Approach B (File-based): render-from-frozen**
- Detect frozen/flattened tracks → copy the frozen audio files directly
- No Live connection needed, but only works on frozen tracks

Start with Approach B since it requires zero Live setup, then add Approach A.

**Files:** `clavus/render.py` (new), `clavus/cli.py` (add `render` command)

### 3. `clavus cue render-als` — Export cues as Live markers

Already partially implemented in `cues.py` (`render_cues_as_markers`). Needs CLI wiring.

- Reads unresolved/active cues from the store
- Generates a `.als`-compatible marker XML fragment
- Inserts into the project file or outputs as a separate marker file

**Files:** `clavus/cli.py` (wire `cue render-als` subcommand)

### 4. Polish & hardening

- `clavus status` should show cues count + dirty state + last snapshot age
- `clavus log --oneline` compact format
- Better error messages when not in a clavus project
- Colorized output

## Proposed Order

| Step | What | Why |
|------|------|-----|
| 1 | `clavus watch` daemon | Most immediately useful — auto versioning |
| 2 | `clavus cue render-als` | Low-effort, already have the parser/cues |
| 3 | `clavus render` (freeze-based) | Useful for collaborators without Live |
| 4 | Polish CLI output | Quality of life |

## Tests

- Watch: create .als, modify mtime, verify snapshot taken
- Render (freeze): create mock frozen track, verify file paths resolved
- CLI: test each new command parses correctly, errors gracefully

## Risks

- **Watch cooldown:** 30s debounce may miss rapid successive saves. Consider configurable.
- **Render (Live OSC):** Requires Live to be running + control surface. Document as optional.
- **Render (freeze):** Only works on frozen/flattened tracks. OK for first pass.
